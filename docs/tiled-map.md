### 1. 总体技术路线（Dual-layer）

本方案把“逻辑占用”与“视觉铺砖”拆成两层：

- **Layer A：逻辑层（Occupancy Layer）**
  - 仅表达某格子是否属于连通区域：`occupied = 0/1`。
  - 负责：连通性计算、填充/擦除、区域 ID 管理、碰撞/寻路（如有）。
  - 不直接追求好看，只保证数据正确。
- **Layer B：视觉层（Visual Layer）**
  - 负责把 Layer A 的占用格子“翻译”为真正的 tile 图片。
  - 通过 **邻域规则 + mask 查表** 决定每个格子使用 tileset 的哪一张（边、角、内填充等）。

> Dual-layer 的核心收益：
>

> 1）逻辑层永远简单稳定（只维护 0/1），不会被美术规则污染。
>

> 2）视觉层规则可随时迭代，甚至支持换 tileset。
>

> 3）当区域变化时，只需局部重算，性能可控。
>

### 2. 数据结构与模块边界（建议）

#### 2.1 OccupancyGrid（逻辑网格）

**职责**：维护占用数据与连通信息。

- 输入来源：
  - 关卡数据导入（初始形状）
  - 玩家操作（涂抹/放置/消除）
  - 生成器（程序生成）
- 对外能力（功能需求）：
  - `setCell(x,y,occupied)`：设置单格占用
  - `getCell(x,y)`：读取占用
  - `getRegionCells(seed)`：从某格获取同一连通块（BFS/DFS）
  - `recomputeRegionIfNeeded(changedCells)`：变化后更新连通信息（可选，视玩法复杂度）

#### 2.2 AutoTileResolver（规则解析器）

**职责**：给定某格坐标 + OccupancyGrid，输出视觉 tile 的“类型/索引”。

- 关键要求：
  - 规则完全数据驱动（mask→tileIndex 表），便于迭代
  - 可切换：4 邻域（N/E/S/W）或 8 邻域（含对角）
  - 约束：输出必须落在“16 张 tileset”的索引集合里

#### 2.3 VisualTilemapRenderer（渲染器）

**职责**：把 AutoTileResolver 的输出写入视觉层 tilemap。

- 功能需求：
  - `refreshAll()`：全量重建（进入关卡、换 tileset、规则大改时）
  - `refreshLocal(changedCells)`：局部刷新（常用）

### 3. 视觉层的核心规则：mask → tile（用 16 张 tileset）

#### 3.1 邻域选择

- **8 邻域 mask（当前实现）**
  - 检查周围 8 个单元格（上/右上/右/右下/下/左下/左/左上）是否 occupied。
  - 原始 mask 为 8 bit（256 种组合），通过**对角线遮罩**和**查表映射**归约到 16 张 tileset。
  - 对角线遮罩规则：对角位仅在两个相邻基数方向均 occupied 时才生效。
  - 适合：块状区域、内凹角/外凸角、精致边界过渡。

#### 3.2 tileset 语义约定（功能需求）

16 张 tile 对应 16 种典型邻域 pattern（详见 `docs/bit_mask.md`）：

- 约定 bit 顺序（顺时针，从上方开始，固定）：
  - bit0 = T (上)
  - bit1 = TR (右上)
  - bit2 = R (右)
  - bit3 = BR (右下)
  - bit4 = B (下)
  - bit5 = BL (左下)
  - bit6 = L (左)
  - bit7 = TL (左上)

- 16 张 tile 语义：

  | Tile | 含义 | 典型 pattern (3×3) |
  |------|------|-------------------|
  | 0 | 孤立格 | 周围无邻居 |
  | 1 | 内部（全填充） | 周围 8 格全占 |
  | 2 | 上边 | 缺 T 方向 |
  | 3 | 左边 | 缺 L 方向 |
  | 4 | 下边 | 缺 B 方向 |
  | 5 | 右边 | 缺 R 方向 |
  | 6 | 内凹角 TL | 缺 TL 对角 |
  | 7 | 内凹角 BL | 缺 BL 对角 |
  | 8 | 内凹角 BR | 缺 BR 对角 |
  | 9 | 内凹角 TR | 缺 TR 对角 |
  | 10 | 外凸角 BR | 仅 R+BR+B |
  | 11 | 外凸角 TR | 仅 T+TR+R |
  | 12 | 外凸角 TL | 仅 L+TL+T |
  | 13 | 外凸角 BL | 仅 B+BL+L |
  | 14 | 双内凹 TL+BR | 缺 TL 和 BR 对角 |
  | 15 | 双内凹 TR+BL | 缺 TR 和 BL 对角 |

- 映射方式：**256 → 16 查表映射**
  - `TileMapConfig.maskTable` 是 256-entry 查表（key: 8-bit raw mask → value: tile index 0–15）
  - 由 `createDefaultMaskTable()` 自动生成：先对角线遮罩，再匹配 canonical pattern

- 对美术/资源的需求：
  - 每张 tile 的视觉边缘要能与邻居无缝拼接（同一风格的边/角）
  - 若有"边缘描边/阴影"，需要保证相邻 tile 的描边不会叠加成双线
  - 若使用图集，需要处理 tile 边缘采样（避免黑边/缝）

### 4. 局部更新策略（运行时性能关键点）

当 OccupancyGrid 发生变化（某格从 0↔1）：

- 必须刷新：
  - 变化格本身
  - 变化格的 8 邻居（上/右上/右/右下/下/左下/左/左上）

**功能需求**：

- 维护一个 `dirtyCells` 集合（Set）
- 每次操作后把"受影响坐标"加入 dirtyCells（自身 + 8 邻居 = 最多 9 个）
- 在一帧末尾或批处理时统一刷新 dirtyCells（避免重复刷）

> 这样能保证：地图再大也不会因为少量编辑导致全图重建。
>

### 5. 连通区域（Connected Region）的定义与获取

你提到“网格中任意的连通区域”，这里建议把“连通”作为逻辑层能力：

- **连通定义（建议默认）**：4 邻域连通（N/E/S/W）
- **功能需求**：
  - 输入：一组 occupied cell 或一个 seed cell
  - 输出：同一连通块的 cell 列表（用于关卡判定、消除、计分、上色等）
  - 复杂度：BFS/DFS 均可

> 注意：
>

> - “连通块计算”与“视觉 mask”可以使用同一套邻域（都用 4 邻域），这样直觉一致。
>

> - 如果玩法需要对角也算连通（8 连通），请明确写入规则，否则视觉与逻辑会出现反直觉。
>

### 6. 配置与可测试性要求（强烈建议）

为了让规则不靠“记忆”，而是可验证、可迭代：

- **mask→tile 映射表配置化**
  - 放在 JSON / Scriptable 配置中
  - 配置里写清楚 bit 顺序与语义
- **可视化调试需求**（开发期）
  - 可在编辑器/调试模式显示某格的 mask 值
  - 可高亮 dirtyCells 的刷新范围
  - 可一键跑“16 种 mask 的预览网格”（快速校验 tileset 是否对齐）

---

### 7. 最小可用验收（MVP Checklist）

- 能从关卡数据生成一块任意形状的 occupied 区域，并正确渲染
- 对任意单格 add/remove 操作：视觉只局部更新且边界正确
- 支持至少两种典型形状：
  - 大块填充（内部块 + 外边界）
  - 细长通路（直线、拐角、T 字、十字）
- 连通块查询结果正确（4 连通），并能用于你的玩法判定

---

### 8. 后续扩展（先记，不做也不影响 MVP）

- 第三层 overlay：专门补“内凹角/装饰角”（在只有 16 张 tile 的情况下提升精致度）

- 运行时批量编辑：笔刷、矩形填充、洪泛填充，并做合并 dirtyCells 刷新
